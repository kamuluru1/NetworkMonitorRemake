# Network Monitor — File Roadmap

A file-by-file plan for the **capture core** (Phases 1–2 of the project: capture
packets off the wire on one thread, process them on a pool of workers). It is
laid out so the later phases — Postgres storage, gRPC/REST services, anomaly
detection — slot in later without having to rework what you've built.

**Four design rules everything below follows**

- **One responsibility per file.** If a file is doing two jobs, it's two files.
- **Dependencies flow one direction.** Everything ultimately rests on
  `packet.hpp`; nothing depends on `main`. No cycles, ever.
- **libpcap is sealed inside one `.cpp`.** The rest of the program never sees
  `pcap.h`, so swapping the capture backend later touches exactly one file.
- **Reusable code lives in a library; executables are thin shells.** The app and
  the test runner are both small `main`s that link the same library code.

## The files

### 1. `include/netmon/packet.hpp` — the data type

The single vocabulary type the whole system passes around. It holds one captured
packet: a timestamp of when the kernel saw it, the original length the packet had
on the wire, and an owned buffer of the captured bytes ("owned" is the key word —
the bytes are copied in, so the packet stays valid after libpcap reuses its
buffer).

It is **header-only** — there's nothing to implement, it's plain data. It depends
only on the standard library (chrono, a fixed-width integer type, and a byte
container). Keep it free of any project dependency, because it sits at the bottom
of the stack and everything else includes it.

### 2. `include/netmon/packet_queue.hpp` + `src/packet_queue.cpp` — the buffer

The bounded, thread-safe hand-off between the one producer and the worker pool.

The **header** holds the class definition: the four public operations — enqueue a
packet (reporting back whether it was accepted or dropped because the queue was
full), dequeue one (blocking until a packet is available or until shutdown), a
shutdown signal, and a read-only accessor for the dropped count — plus the private
members: a mutex, a condition variable, the underlying standard queue, the
capacity limit, a "done" flag, and an atomic dropped counter.

The **`.cpp`** holds the bodies: the locking, the bounded-drop logic, the
condition-variable wait, and the drain-then-exit shutdown behavior. It depends on
`packet.hpp` (it stores packets) and standard threading headers.

*Note:* this class is small enough that it could legitimately be header-only, but
splitting it is the instructive default and keeps the example consistent. It is
also the most testable piece, which is why it's the first thing to write and the
first thing to test.

### 3. `include/netmon/capturer.hpp` + `src/capturer.cpp` — the producer

Owns the libpcap handle and runs the capture loop. **This is the only file in the
whole project that knows libpcap exists.**

The **header** declares a `Capturer` class: it is constructed from a device name
plus capture settings (opening the device, and reporting failure if it can't),
exposes a method that runs the capture loop — taking a reference to the queue and
a reference to the "running" flag, and pushing copied packets until told to stop —
and a method to stop it. Its destructor closes the handle automatically (RAII), so
nobody ever has to remember to clean up. Critically, the header does **not**
include `pcap.h`; it hides the handle behind an opaque pointer so the libpcap
dependency doesn't leak to every file that includes the capturer.

The **`.cpp`** is where `pcap.h` is included. It opens and closes the device, runs
the capture loop (fetch packet, **copy the bytes** into a packet, push to the
queue, count), and translates libpcap errors into your own error type.

It depends on `packet.hpp` and `packet_queue.hpp`; libpcap stays internal. This
replaces the loose capture function and the `g_handle` global from the starter
file — and because it's isolated, the day you move to PcapPlusPlus or raw sockets,
only this one file changes.

### 4. `include/netmon/worker.hpp` + `src/worker.cpp` — the consumer

The other side of the queue: pull packets out and process them. This is the seam
where every future feature lands.

The **header** declares the worker routine: it takes a reference to the queue and
a reference to the run's counters, and loops pulling packets until the queue is
shut down and drained.

The **`.cpp`** holds that loop. Today it does the minimum — count each processed
packet. Tomorrow this is where you parse the Ethernet/IP/TCP headers, update
per-flow state, and hand findings to the detection layer. It depends on
`packet.hpp`, `packet_queue.hpp`, and `stats.hpp`.

*Note:* capture and worker shared a file in the starter, but they're genuinely
different jobs (producer vs consumer) that grow in different directions, so they
get separate files now.

### 5. `include/netmon/stats.hpp` — the run's counters

A small struct that bundles the atomic counters for one run (packets captured,
packets processed) so you pass one object around instead of several loose atomic
references. It is **header-only** and depends only on the standard atomics. Minor
on its own, but it keeps function signatures clean as the program grows.

### 6. `include/netmon/config.hpp` — the settings

One home for every tunable, plus the logic to build it from the command line. It
holds a `Config` struct — interface name, snap length, promiscuous flag, read
timeout, queue capacity, worker count — and a small function that turns the
command-line arguments into a `Config` with sensible defaults. **Header-only** (or
a tiny `.cpp` if argument parsing grows), depending only on the standard library.
This removes the magic numbers currently sitting inline in `main`.

### 7. `src/main.cpp` — the one entry point

Assembles the pieces and owns the program's lifecycle; nothing reusable lives
here. It contains `main()`: build the config from the arguments, construct the
queue and the capturer, install the signal handler, spawn the one capture thread
and the worker pool, wait for shutdown, join everything, and print the final
counts. The signal handler and the single "running" flag also live here — the flag
is **owned here and passed by reference into the capturer**, so there's no global
shared across files (the fix for the entanglement in the starter).

It depends on every header above and is the top of the stack — nothing includes
`main`. Keep it thin: if `main` gets long, logic that belongs in a component has
leaked upward.

### 8. `CMakeLists.txt` — the build wiring

Encodes the "shared library + thin executables" structure. It builds a **library
target** (e.g. `netmon`) from the `src` implementations (queue, capturer, worker);
a **main executable** (`network_monitor`) from `main.cpp`, linked against the
`netmon` library plus libpcap and the threads library; and a **test executable**
from the `tests` files, linked against `netmon` plus GoogleTest. It locates
libpcap and the threads package. This is where "multiple mains = separate
executables over shared code" becomes concrete: the app and the test runner both
link the same library.

### 9. `tests/test_packet_queue.cpp` — the first tests

Proves the queue obeys the rules you designed for it. Cases: push then pop returns
the same packets in order; filling past capacity rejects the extra and bumps the
dropped counter; after shutdown, a drained pop returns false. Its `main()` is
supplied by GoogleTest — the concrete example of "another main, a separate
executable."

It depends on `packet_queue.hpp`, `packet.hpp`, and the test framework. The queue
is testable with **zero network hardware**, which is exactly why it's first.
Capturer and worker tests come later and feed in a saved `.pcap` sample file
rather than a live interface.

### 10. `README.md` and `.gitignore`

`README.md`: what the project is, how to build it, how to run it, and the note
that live capture needs `sudo`. `.gitignore`: ignore `build/` and editor cruft so
generated files never get committed.

## Build order

Write them bottom-up so every file you create can compile and be tested against
something that already works:

1. **`packet.hpp`** — the type everything else needs.
2. **`packet_queue.hpp` + `.cpp`**, then **`test_packet_queue.cpp`** — build the
   buffer and prove it works before anything depends on it.
3. **`config.hpp`** and **`stats.hpp`** — small, dependency-light plumbing.
4. **`capturer.hpp` + `.cpp`** — the libpcap producer, against the finished queue.
5. **`worker.hpp` + `.cpp`** — the consumer.
6. **`main.cpp`** — wire it all together last.
7. **`CMakeLists.txt`** — grow it as you add each target (library first, then the
   executable, then the test runner).

## Where the later phases attach

The structure is designed so these are additions, not rewrites:

- **Storage (Phase 3)** — a new `src/storage/` module with a `DbWriter` (libpqxx)
  that drains a second queue into Postgres via `COPY`. The worker hands it
  finished records; nothing existing changes shape.
- **gRPC + REST (Phase 4)** — a `proto/` directory for the contract and generated
  code, plus a service wrapper. This is where the project splits into two
  executables (capture service, analytics service), each its own thin `main` over
  the shared library.
- **Detection (Phase 5)** — a `src/detection/` module of analyzers (protocol
  mismatch, flash crowd, DDoS) that the worker calls into. It consumes flow state
  and emits events; it doesn't touch capture or storage.
